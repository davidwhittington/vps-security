1. Overview & Architecture

This guide addresses the most common security gaps found on a fresh or lightly configured VPS running Ubuntu with Apache. The hardening is done in five phases, each with a corresponding shell script in the scripts/ directory.

ScriptPurposeRisk Level
01-immediate-hardening.shFirewall, fail2ban, SSH lockdown, sysctlCritical
02-apache-hardening.shSecurity headers, info disclosure, mod_statusHigh

04-monthly-updates-setup.shAutomated monthly apt upgrade with email reportMedium
05-log-monitoring-setup.shLogwatch daily digest + GoAccess traffic reportsLow
Important: Always run scripts in order. Phase 1 must be completed first — it addresses actively exploitable vulnerabilities. Each script includes pre-flight checks and will abort on errors.

3. Phase 1 — Immediate Hardening

Script: scripts/01-immediate-hardening.sh

Run this first. On a typical unconfigured VPS, the server is actively being brute-forced within minutes of going online. This script addresses the three most critical gaps.

3a. Install & Configure Fail2Ban

Blocks IPs after repeated failed SSH login attempts.

apt install -y fail2ban

cat > /etc/fail2ban/jail.local << 'EOF'
[DEFAULT]
bantime = 3600
findtime = 600
maxretry = 3
banaction = iptables-multiport

[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
EOF

systemctl enable --now fail2ban

3b. Enable UFW Firewall

Restricts inbound traffic to SSH, HTTP, and HTTPS only.

ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp    # SSH
ufw allow 80/tcp    # HTTP
ufw allow 443/tcp   # HTTPS
ufw --force enable

3c. Harden SSH

Disable password auth, restrict root to key-only, disable X11 forwarding.

# Disable password authentication
sed -i 's/^PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
echo 'PasswordAuthentication no' > /etc/ssh/sshd_config.d/50-cloud-init.conf

# Root login: key-only (use "no" to fully disable)
sed -i 's/^PermitRootLogin yes/PermitRootLogin prohibit-password/' /etc/ssh/sshd_config

# Disable X11 forwarding
sed -i 's/^X11Forwarding yes/X11Forwarding no/' /etc/ssh/sshd_config

# Ubuntu 24.04 uses "ssh" not "sshd"
systemctl reload ssh
Before reloading SSH: Verify your SSH key is in /root/.ssh/authorized_keys. Test in a second terminal before closing your current session.

3d. Kernel Network Hardening

cat > /etc/sysctl.d/99-hardening.conf << 'EOF'
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.default.accept_redirects = 0
net.ipv4.conf.all.send_redirects = 0
net.ipv4.conf.default.send_redirects = 0
net.ipv6.conf.all.accept_redirects = 0
net.ipv6.conf.default.accept_redirects = 0
net.ipv4.conf.all.log_martians = 1
net.ipv4.conf.default.log_martians = 1
EOF

sysctl --system

4. Phase 2 — Apache Hardening

Script: scripts/02-apache-hardening.sh

4a. Enable mod_headers

a2enmod headers

4b. Update security.conf

cat > /etc/apache2/conf-enabled/security.conf << 'EOF'
ServerTokens Prod
ServerSignature Off
TraceEnable Off

# Block version control directories
RedirectMatch 404 /\.git
RedirectMatch 404 /\.svn

<IfModule mod_headers.c>
    Header always set X-Content-Type-Options "nosniff"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"
    Header always set Permissions-Policy "geolocation=(), microphone=(), camera=()"
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
    Header always set X-XSS-Protection "0"

    # CSP frame-ancestors: allow your own domains to iframe each other.
    # Customize this list for each server deployment.
    Header always set Content-Security-Policy "frame-ancestors 'self' *.yourdomain.com yourdomain.com otherdomain.com"
</IfModule>
EOF
iframe Note: We use Content-Security-Policy: frame-ancestors instead of X-Frame-Options because it supports multiple domains. List all domains that need to embed each other. This was necessary because davidwhittington.com iframes sites like ipvegan.com.

4c. Disable mod_status

a2dismod status

4d. Test & Reload

apache2ctl configtest && systemctl reload apache2

5. Phase 3 — Admin User Setup

Script: scripts/03-setup-admin-user.sh

Creates a proper admin user with sudo access so you don't need to SSH as root for daily tasks.

# Update shell and add to sudo group
usermod -s /bin/bash USERNAME
usermod -aG sudo USERNAME

# Copy SSH keys from root
mkdir -p /home/USERNAME/.ssh
cp /root/.ssh/authorized_keys /home/USERNAME/.ssh/
chown -R USERNAME:USERNAME /home/USERNAME/.ssh
chmod 700 /home/USERNAME/.ssh
chmod 600 /home/USERNAME/.ssh/authorized_keys

# Remove cloud-init NOPASSWD rule
rm -f /etc/sudoers.d/90-cloud-init-users

6. Phase 4 — Automated Updates & Email Reports

Script: scripts/04-monthly-updates-setup.sh

6a. SMTP Relay via msmtp

Install msmtp and configure it to send email through an external SMTP account.

apt install -y msmtp msmtp-mta mailutils

cat > /etc/msmtprc << 'EOF'
defaults
auth           on
tls            on
tls_trust_file /etc/ssl/certs/ca-certificates.crt
logfile        /var/log/msmtp.log

account        default
host           mail.privateemail.com
port           587
from           sender@yourdomain.com
user           sender@yourdomain.com
password       REDACTED

EOF

chmod 600 /etc/msmtprc
SMTP Provider Notes: Namecheap Private Email works with regular account passwords. App passwords generated for CalDAV/CardDAV/Backup do not authorize SMTP. Microsoft 365 managed accounts may not allow app passwords — use a provider you control.

6b. Monthly Update Report

A cron job at 0 3 1 * * (3 AM UTC, 1st of month) runs apt upgrade and emails a report covering:

  • Packages upgraded and any remaining
  • Kernel version check (reboot needed?)
  • Disk usage
  • Fail2ban status and banned IPs
  • TLS certificate expiry dates

7. Phase 5 — Log Monitoring

Script: scripts/05-log-monitoring-setup.sh

7a. Logwatch — Daily Email Digest

Parses all system logs and emails a summary. Runs automatically via /etc/cron.daily/.

apt install -y logwatch

cat > /etc/logwatch/conf/logwatch.conf << 'EOF'
Output = mail
MailTo = recipient@yourdomain.com
MailFrom = sender@yourdomain.com
Detail = High
Range = yesterday
Service = All
Format = html
mailer = "/usr/bin/msmtp"
EOF

Logwatch covers: SSH auth, Apache, fail2ban bans, dpkg installs, sudo usage, disk space, and more.

7b. GoAccess — Apache Traffic Reports

Generates an interactive HTML dashboard from Apache access logs. Runs daily at 4 AM UTC.

apt install -y goaccess apache2-utils

# Daily report generation
cat $LOGS | goaccess \
    --log-format=COMBINED \
    --no-global-config \
    --output="/var/www/html/reports/traffic-report.html"

The report directory is protected with HTTP basic auth:

htpasswd -cb /etc/apache2/.htpasswd-reports admin "GENERATED_PASSWORD"

# /var/www/html/reports/.htaccess
AuthType Basic
AuthName "Server Reports"
AuthUserFile /etc/apache2/.htpasswd-reports
Require valid-user

7c. Future Consideration: CrowdSec

For servers needing more active protection, consider CrowdSec as a fail2ban replacement. It adds crowdsourced IP blocklists (~50 MB RAM). Evaluate on a per-server basis — don't run both simultaneously.

10. Full Hardening Checklist

Use this checklist for each new server deployment:

Critical — Do Immediately

  • Install and enable fail2ban with sshd jail
  • Enable UFW firewall (allow 22, 80, 443 only)
  • Disable SSH password authentication
  • Set SSH root login to prohibit-password or no

High — Do Same Day

  • Enable mod_headers in Apache
  • Set ServerTokens Prod and ServerSignature Off
  • Add security headers (HSTS, X-Content-Type-Options, CSP frame-ancestors, Referrer-Policy)
  • Block .git / .svn access in Apache
  • Disable mod_status or restrict to localhost
  • Add Options -Indexes to all virtual hosts

Medium — Do This Week

  • Apply all pending system updates (apt upgrade)
  • Harden sysctl network parameters
  • Set up dedicated admin user with sudo
  • Disable X11Forwarding in SSH
  • Remove cloud-init NOPASSWD sudoers rule
  • Configure SMTP relay for server email
  • Set up monthly update cron with email report

Low — Do This Month

  • Install Logwatch for daily log digests
  • Install GoAccess for traffic monitoring
  • Verify certbot auto-renewal (certbot renew --dry-run)
  • Consider changing SSH port (reduces noise)
  • Review and reboot if newer kernel is available
  • Evaluate CrowdSec as fail2ban replacement

11. Quick Reference Commands

Check Security Status

# Firewall
ufw status

# Fail2ban — currently banned IPs
fail2ban-client status sshd

# Recent failed SSH logins
grep "Failed password" /var/log/auth.log | tail -20

# Apache security headers
curl -sI https://yourdomain.com | grep -iE "server:|x-content|strict-transport|content-security|referrer"

# Certificate expiry
certbot certificates

# Pending updates
apt list --upgradable

# Kernel check
uname -r
ls /boot/vmlinuz-* | sort -V | tail -1

Service Management

# SSH (Ubuntu 24.04 uses "ssh" not "sshd")
systemctl reload ssh
systemctl status ssh

# Apache
apache2ctl configtest
systemctl reload apache2

# Fail2ban
fail2ban-client status
fail2ban-client set sshd unbanip 1.2.3.4

# UFW
ufw status numbered
ufw delete RULE_NUMBER
